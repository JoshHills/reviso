CSS: Basics

Ready your palettes for some pretty-picture-pilates, as we explore how to add an almost essential building block to our websites in the effort to spruce them up with a lick of pristine paint! Cascading StyleSheets.css allow you to select one or more of your HMTL elements, and provide rules for how they should be displayed on the screen- this ranges from physical modifications such as size and position, to aesthetic tweaks such as color, rounding and shadows. Covering the basics, we will look at exactly how this format came into being, how it works, and some basic implementations.

CSS 'cascades' down a page of style rules, linking relevant aesthetic instructions with elements.
Clients then do their best to render these rules as faithfully as possible.
There exists a venerable onslaught of complicated modifiers you can use to adapt your styling.

Pizazz!

HTML on its own is rendered in an undeniably dull demeanour. CSS allows us to separate our style from our structure, which meets the needs of that design ethos we talked about- a ‘separation of concerns’. In this way, we can take an HTML document, and give it an infinite amount of different designs- this is particularly useful as it allows for easy, on-the-fly adjustments to presentation, and means that we can freshen up our web-pages at any time without necessarily altering their core structure or contentlike redesigning your bedroom, or something.

Style sheets specify formatting rules which consist of selectors and declarations: in English, you may stateif you think my jokes are bad, by the way, wait til you see these- relevancy! “I want my paragraphs to be the colour of a blanched almond”, and this translates precisely to:

p { color: BlanchedAlmond; }

Connecting the dots:

Stylesheets can be incorporated into your HTML in a multitude of ways: most simplistically, a <style> tag can be employed which reserves a namespace for your CSS rules. These must appear nested within the <head> tag, as this is loaded before the rest of the DOM elements on the page- the browser will want to know how to display elements before actually trying it!

Inline Styling, it's not you, it's me...

Ok. Ok. So.. ok. Inline<p style="color: BlanchedAlmond"> styling is generally not a great idea for the same reason that nested styling in style tags also mightn’t be a good idea. They clutter the code, and defeat the purpose of CSS, which is to provide re-usable, blanket styling to classed or tagged elements. By being so specific, you make it harder to change the layout of your website at a larger scale. Don’t do this. Please, think of the children.

As a disclaimer, when you modify styles with JavaScript when the page has already loaded, you usually add/remove inline styles- this is okay, as those changes are generally flushed upon reload and do not alter any source code. Selectors:

Now that we're solidly grounded, it's time for the biggest, best-est bit of CSS; part of being a web-whiz is being able to navigate fluidly through the tree-structure of nodes that comprises your HTML document. This is achieved by almost charting your path through the nodes, separated by spaces, to get to the target destination. This selection path can be fiddled with, but in its basic form, it may look something like this:

element-name | #element-id | .class { style: rule; }

By only writing an element’s name, you select all instances of that element to style. By using an ID, you select the element with that ID, and by applying style rules to a class (which is what they were made for!) you style any elements classed as such. The path reads from left to right, following a string representation of the nesting of the tree structure- the more elements you provide, the deeper you're going. For example, to select an anchors that are within a paragraph that is within a divider, you'd use: div p a { ... }.

Operators:

+selects all elements immediately following an element
~selects every element preceded by an element
>selects all direct children
*selects all elements

To fine-tune your selectionsand essentially perform syntactical backflips to an even greater degree, CSS provides some essential modifiers to the path selection, as well as things called 'pseudo-classes' and even some functions- these are recent developments as a result of CSS3's release, and open some interesting, if a little peculiar opportunities.

Prevalence:

In CSS, as the document cascades downwardsonly once!, style rules applying to the same elements will over-write each-other. Understanding this, you should know that CSS favours style rules declared last, however it also favours specificity, which means that if a rule comes later but is less specific than a previous one, it will not be considered more importantyou can actually force a rule to the forefront by appending !important to your rule, however this should be used sparingly for fringe-cases.. The developer tools of most browsers display rules that are being actively overridden, and allow you to toggle them respectively.
